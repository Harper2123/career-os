# Career OS Setup Roadmap

This roadmap is executed through explicit gates. A step begins only after the command `Proceed to Step X` is given. Each step ends with verification before the next begins.

## Step 1 — Establish the central source of truth

Create and verify the `career-os` repository, governing documents, live state file, long-term direction, setup roadmap, and first decision record.

**Completion condition:** the repository clearly answers what Career OS is, what is current, why the system exists, and what comes next.

## Step 2 — Configure the ChatGPT workspace

Define the minimum ChatGPT Projects and conversations, shared context rules, handoff conventions, completion summaries, and boundaries between planning, MScFE, projects, and public presence.

**Completion condition:** every major type of conversation has a clear home and important state is written back to Career OS.

## Step 3 — Prepare WSL and Docker as the primary development environment

Verify WSL, configure the Linux project directory, Git, shell tools, Docker Desktop's WSL 2 backend, Ubuntu Docker integration, and connectivity from Windows Visual Studio Code. Substantial Python project runtimes and dependencies are isolated through repository-owned container definitions rather than direct project `venv`, Conda, or Miniforge environments.

**Completion condition:** a test Python project stored in the WSL Linux filesystem builds and runs through Docker from Ubuntu, then opens successfully through VS Code connected to WSL and, where appropriate, a Dev Container.

## Step 4 — Configure VS Code as the engineering control centre

Set up the minimum useful profiles and extensions for Python, Jupyter, Markdown, testing, GitHub, Docker, WSL, and Dev Containers. Keep automatic AI completion disabled by default.

**Completion condition:** Python, notebooks, Markdown, tests, Git operations, terminal commands, and containerized development work consistently from VS Code using WSL.

## Step 5 — Establish the Git and GitHub workflow

Define and practise issue selection, branch naming, Conventional Commits, draft pull requests, review, revisions, merge strategy, debt recording, and issue closure.

**Completion condition:** one harmless practice issue completes the full workflow.

## Step 6 — Set up the Obsidian knowledge system

Create a lightweight vault for concepts, finance, AI engineering, mathematics, MScFE, books, research, and maps of content. Do not add LLM integration in v1.

**Completion condition:** one completed MScFE concept is captured, connected, and retrievable.

## Step 7 — Connect Calendar, Tasks, Gmail, and private storage

Assign fixed deadlines to Calendar, introduce only the minimum useful scheduled reminders, define Gmail's role for announcements, and organise private course resources.

**Completion condition:** one end-to-end deadline or communication workflow succeeds without creating competing task systems.

## Step 8 — Build the MScFE module workflow

Create templates for material inventory, module brief, question map, safe assignment support, notebook-to-module progression, and module closure.

**Completion condition:** the workflow is demonstrated on one completed module before being used for the next live course.

## Step 9 — Design the project and research system

Define project-selection criteria, supporting lab rules, portfolio-quality standards, research-to-engineering progression, and project lifecycle. Do not start a large project merely to populate the system.

**Completion condition:** a selection framework exists and no premature project has been started.

## Step 10 — Create the sustainable public-presence workflow

Define how meaningful work flows from GitHub to optional Medium articles, concise LinkedIn insights, and curated portfolio entries.

**Completion condition:** a publication pipeline and backlog structure exist without a posting quota.

## Step 11 — Validate Career OS with a real work cycle

Run one short cycle from task selection through work, documentation, review, and weekly reflection. Modify the system only where observed friction justifies it.

**Completion condition:** one complete cycle succeeds without harming sleep or creating guilt debt.

---

## Governing setup rule

The environment is successful when it is reliable enough to support real work. It does not need to be perfect before learning and engineering begin.
